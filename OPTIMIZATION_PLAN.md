# 优化与重构计划

> 目标：在不改变现有业务能力的前提下，优先解决安全、性能、可维护性和回归保护问题。

## 现状摘要

- 后端核心入口 `main.py` 体量较大，当前已经接近 5000 行。
- 前端群组详情页和专栏页也已明显膨胀，单文件包含大量状态与请求逻辑。
- 配置文件中存在真实敏感信息，属于最高优先级风险。
- SQLite 查询存在可优化空间，尤其是评论图片的 N+1 查询。
- 任务状态和日志主要依赖内存全局变量，重启即丢失。

## 优先级路线

### Phase 1: 安全与稳定性

1. 移除仓库中的真实敏感信息，改为环境变量或本机私密配置。
2. 收紧 CORS 配置，避免 `*` + `allow_credentials=true` 的组合。
3. 清理生产环境中的调试日志输出。

### Phase 2: 性能与数据库

1. 为高频过滤字段补充 SQLite 索引。
2. 消除评论图片读取的 N+1 查询。
3. 识别并减少高频页面的重复请求。

### Phase 3: 架构拆分

1. 将 `main.py` 按业务域拆分路由。
2. 将后台任务执行逻辑从路由中剥离到 service 层。
3. 封装任务状态/日志管理器，统一内存状态入口。

### Phase 4: 前端收敛

1. 拆分超大的群组详情页和专栏页。
2. 拆分 `api.ts`，按领域组织 API client。
3. 降低任务轮询频率，统一任务监控策略。

### Phase 5: 回归保护

1. 为核心 API 补最小测试集。
2. 增加基础 CI 校验：语法、单测、前端 lint/build。

## 并行执行分工

### Agent A

- 负责安全与配置相关改动。
- 目标文件优先级：
  - `config.toml`
  - `main.py`
  - `README.md`
- 重点任务：
  - 去除仓库中的真实密钥
  - 增加/调整配置读取方式
  - 收紧跨域配置

### Agent B

- 负责数据库性能与查询优化。
- 目标文件优先级：
  - `zsxq_database.py`
- 重点任务：
  - 添加必要索引
  - 优化评论图片读取逻辑
  - 保持现有返回结构不变

### Agent C

- 负责前端请求层和任务监控的收敛。
- 目标文件优先级：
  - `frontend/src/lib/api.ts`
  - `frontend/src/components/TaskPanel.tsx`
  - `frontend/src/components/TaskLogViewer.tsx`
- 重点任务：
  - 移除调试日志
  - 降低轮询噪音
  - 保持现有交互行为

## 执行原则

- 每个 agent 只修改自己的责任文件，避免交叉覆盖。
- 不要回滚别人已经完成的改动。
- 保持最小 diff，不做无关重构。
- 完成后优先跑最小可验证检查。
